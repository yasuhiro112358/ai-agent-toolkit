# 解説: 設計の判断

## なぜ同一セッションで実行するか

ステップ1のプロンプトは、作業を行った **同じ会話セッション** の末尾に貼り付けて実行する設計になっている。「過去の会話をコピーして新しいセッションに貼り付ける」方式は採用しなかった。

**同一セッション実行の利点:**
- コンテキストをコピーする手間がない（貼り付けるだけで即座に実行できる）
- 会話の全文が AI のコンテキストウィンドウに既に存在するため、抽出漏れが起きにくい
- 会話中のニュアンス（口調・前提・背景）がそのまま分析に反映される

**「コンテキストを別途貼る」方式の問題:**
- コピーする範囲の判断が利用者に委ねられ、重要な文脈が抜け落ちるリスクがある
- UI によっては会話の全文コピーが難しい

## なぜ直接生成ではなく 2 ステップ構成か

「会話 → プロンプト」の 1 ステップ直接生成は採用しなかった。

**直接生成の問題:**
- 会話の表面的な「何をしたか」に引きずられ、「なぜしたか」が失われやすい
- 固有値（プロジェクト名・特定のコード）がそのままプロンプトに残り、汎用性が低下する
- 出力の検証が難しい（何が正しいプロンプトかの判断基準が明確でない）

**2 ステップの利点:**
- 要件定義書が中間成果物として人間のレビューポイントになる
- ステップ1の出力（要件定義書）はそのままステップ2の入力として使える
- 要件定義書の品質が最終プロンプトの品質を決定するため、修正の効果が見えやすい

## なぜ要件定義書を中間層に置くか

要件定義書（IEEE 830 SRS-lite 形式）を中間層に置く理由は、「何をするか」と「なぜするか」を分離するためである。

| 問い | 担当する層 |
|------|-----------|
| なぜこのタスクが必要か | ゴール・アクター（要件定義書） |
| 何をするか | タスク概要・入力仕様・出力仕様（要件定義書） |
| どう指示するか | プロンプト（ステップ2の出力） |

この分離により、同じ要件定義書から異なるトーン・形式のプロンプトを生成することも可能になる。

## なぜ自己チェック＋修正を要件に含めるか

初期設計では「確認する」のみだったが、確認だけでは利用者が追加の修正指示を出す必要があり、UI 上での往復が増えていた。

**自己チェック＋修正を採用した理由:**
- AI が問題を発見したその場で修正することで、利用者に届く成果物の品質が担保される
- 「チェック → 修正 → 出力」を 1 往復で完結させることで、利用者の操作を最小化できる
- 自己チェック項目が明示されているため、何を基準に修正されたかが利用者に伝わる

## なぜ特定の AI サービスに依存しないか

生成 AI のサービスは急速に変化する。特定サービスへの依存は、サービス仕様変更・料金改定・廃止のリスクを抱える。

**移植性を確保するための制約:**
- API・外部ツール不使用（プロンプトを貼り付けるだけで動作する）
- Mermaid 等の特定ツール形式に依存しない出力
- Claude / ChatGPT / Gemini 等の主要 UI で同等の結果が得られること

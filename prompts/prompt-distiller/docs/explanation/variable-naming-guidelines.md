# 解説: 変数設計のガイドライン

[ドキュメント](../index.md) > 変数設計のガイドライン

> **Diataxis: Explanation** — 変数名の粒度・命名規則・プレースホルダー設計の考え方。


変数は、プロンプトの汎用性を決定する最も重要な要素である。「固有値をプレースホルダーに置き換える」という操作は単純だが、粒度と命名の判断が最終的なプロンプトの使いやすさに直結する。

---

## 適切な粒度

変数は「使う場面ごとに変わる最小単位」で切る。

**細かすぎる例（避ける）:**

```text
[会社名] [部署名] [担当者名] が作成した [ファイル名].[拡張子] を...
```

変数が多すぎると入力の手間が増え、プロンプトが使いにくくなる。

**粗すぎる例（避ける）:**

```text
[対象情報] を分析して...
```

変数が大きすぎると、何を入力すべきか伝わらない。

**適切な例:**

```text
[対象ドキュメント] を分析して...
```

「使う人が自然に一つの値として入力できるもの」を一つの変数にする。

---

## 命名規則

### 内容が伝わる名前をつける

変数名はプレースホルダーとして機能するため、その変数に何を入れるべきかが名前から分かる必要がある。

| 悪い例 | 良い例 | 理由 |
| ------ | ------ | ---- |
| `[変数1]` | `[対象ドキュメント]` | 何を入れるか分からない |
| `[X]` | `[対象読者層]` | 短すぎて意味が伝わらない |
| `[テキスト]` | `[要約対象の記事]` | 汎用的すぎて用途が不明 |

### 日本語・英語どちらでも可

要件定義書で使う言語に合わせる。英語のプロンプト用に使うなら英語変数名でも問題ない。

```text
日本語: [対象ドキュメント]
英語:   [target_document]
```

## Before/After 例

### 例1: コードレビュー用プロンプト

**変数化前（固有値が残っている）:**

```text
Python で書かれた REST API の認証モジュール（auth.py, 約200行）をレビューしてください。
対象プロジェクトは社内決済システムです。
```

**変数化後:**

```text
[プログラミング言語] で書かれた [対象モジュールの説明]（[ファイル名], 約[行数]行）をレビューしてください。
対象プロジェクトは [プロジェクト概要] です。
```

**変数一覧:**

| 変数名 | 説明 |
| ------ | ---- |
| `[プログラミング言語]` | レビュー対象コードの言語（Python、TypeScript 等） |
| `[対象モジュールの説明]` | モジュールの役割を一言で（認証モジュール、データ変換処理 等） |
| `[ファイル名]` | レビュー対象ファイルのファイル名 |
| `[行数]` | ファイルのおおよその行数 |
| `[プロジェクト概要]` | プロジェクトの目的・用途を一言で |

### 例2: ドキュメント要約プロンプト

**変数化前:**

```text
以下の会議議事録を3点に絞って箇条書きで要約してください。
対象: 2024年Q3事業レビュー（経営会議）
```

**変数化後:**

```text
以下の [ドキュメント種別] を [要約点数] 点に絞って箇条書きで要約してください。
対象: [ドキュメント名]
```

**変数一覧:**

| 変数名 | 説明 |
| ------ | ---- |
| `[ドキュメント種別]` | 要約対象の文書種別（会議議事録、仕様書、報告書 等） |
| `[要約点数]` | 要約する箇条書きの点数 |
| `[ドキュメント名]` | 要約対象のドキュメント名・タイトル |

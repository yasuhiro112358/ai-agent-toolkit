# ハウツー: 要件定義書を改良する

ステップ1で AI が生成した要件定義書は、AI の自己チェック・修正を経ているが、会話の文脈によっては精度が不十分な場合がある。ステップ2に進む前に人間がレビューすることで、最終的なプロンプトの品質が向上する。

---

## ゴールが「何をしたか」に留まっている場合

**症状:** ゴールが「〇〇を要約する」「コードをレビューする」のような動詞句になっている。

**修正:** 「なぜその作業が必要だったか」を問いかけ、「〜するために〜する」形式に書き換える。

```
悪い例: コードをレビューする
良い例: レビューコストを下げるために、コードの問題点を網羅的に列挙する
```

---

## アクターが抽象的な場合

**症状:** アクターが「エンジニア」「ユーザー」のような一言で終わっている。

**修正:** 役割・専門知識レベル・利用シーンの3点を補足する。

```
悪い例: エンジニア
良い例: バックエンドエンジニア（Python 経験3年以上）。コードレビュー依頼が集中する時間帯に、
         複数 PR を効率よく確認するために使う。
```

---

## 変数化が不十分な場合

**症状:** 要件定義書に特定のプロジェクト名・ファイル名・固有の数値が残っている。

**修正:** 固有値を `[変数名]` 形式のプレースホルダーに置き換え、変数一覧に追加する。

```
変更前: target_audience は 30〜40代の会社員
変更後: target_audience は [対象読者層]

変数一覧に追加:
| `[対象読者層]` | プロンプトを適用する読者の属性（年代・職業等） |
```

---

## 制約条件に暗黙のルールが抜けている場合

**症状:** 明示的に指定したルールだけが書かれており、会話の文脈から自明だった条件が抜けている。

**修正:** 会話を振り返り、「当然の前提」として扱っていた条件を補足する。よくある抜け漏れ:

- 出力言語（日本語で出力すること、など）
- トーン（敬語・箇条書き・Markdown 形式、など）
- 禁止事項（固有名詞を使わない、独自の結論を加えない、など）

---

## 修正後の確認

修正が終わったら、以下のチェックリストで最終確認する:

- [ ] ゴールは「〜するために〜する」形式か
- [ ] アクターに役割・知識レベル・利用シーンが含まれているか
- [ ] タスク概要が一言で明確に表現されているか
- [ ] 入力仕様・出力仕様に形式・構造・トーンが具体的に書かれているか
- [ ] 制約条件に暗黙のルールも含まれているか
- [ ] 変数一覧の固有値がすべて `[変数名]` 形式になっているか

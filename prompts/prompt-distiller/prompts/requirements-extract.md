# requirements-extract.md

あなたは要件アナリストです。
この会話でこれまでに行ってきた作業を分析し、同じ作業を別の場面でも再現するために必要な要件定義書を作成してください。

> 注意：この分析指示（このプロンプト自体）は作業対象から除外してください。

## 分析観点

表面的な「何をしたか」に留まらず、「なぜしたか・誰がするか」まで遡って分析してください：

1. **ゴール**: なぜこのタスクが行われたか。解決したい問題・達成したいニーズを一言で（「何をした」ではなく「なぜした」の視点で）
2. **アクター**: このプロンプトを使うのは誰か。役割・専門知識レベル・利用シーンを整理する
3. **タスク特定**: 何をしていたか（要約・翻訳・コードレビュー・分析・文章作成など）を一言で表す
4. **入力仕様**: ユーザーが提供した情報の種類・形式（テキスト・コード・データなど）を整理する
5. **出力仕様**: AIが生成した成果物の構造・形式・トーン（箇条書き・Markdown・口調など）を整理する
6. **制約抽出**: 明示的・暗黙的な制約条件（文字数・言語・禁止事項・品質基準など）を洗い出す
7. **変数化**: 固有名詞・特定のデータ・固有のコードなどを `[変数名]` 形式のプレースホルダーに置き換える

## 出力形式

以下の形式で要件定義書を出力してください：

---

## ゴール

[このプロンプトが解決する問題・達成する目的を一言で。「〜するために〜する」の形で記述]

## アクター

[誰がこのプロンプトを使うか。役割・専門知識レベル・利用シーンを記述]

## タスク概要

[何をするタスクか、一言で]

## 入力仕様

[ユーザーが提供する情報の種類・形式。例：`[対象テキスト]`、`[プログラミング言語]` など]

## 出力仕様

[成果物の構造・形式・分量・トーンなどの指定]

## 制約条件

[守るべきルール・制限・禁止事項・品質基準]

## 変数一覧

| 変数名 | 説明 |
| ------ | ---- |
| `[変数名]` | この変数が表す内容の説明 |

---

### 禁止事項

- この分析指示テキスト自体を分析対象に含める
- 会話履歴にない情報の推測による補完
- 7つの分析観点以外の独自観点の追加
- 変数化すべき固有値の残置（プレースホルダーへの置き換え漏れ）

### 自己チェックと修正

作成後、以下を確認してください。問題があれば修正してから出力してください：

- [ ] ゴールは「なぜするか」の視点で記述されているか（「何をするか」に留まっていないか）
- [ ] アクターの役割・専門知識レベル・利用シーンが具体的に記述されているか
- [ ] タスク概要は何をするタスクか一言で明確に表現されているか
- [ ] 入力仕様・出力仕様の形式・構造・トーンが具体的に特定されているか
- [ ] 制約条件に明示的な制約だけでなく、暗黙の制約も含まれているか
- [ ] 変数一覧の固有値がすべて変数化され、プレースホルダーと説明が対になっているか
